## 1. 执行摘要

### 1.1 结论

Infinite Canvas 2 **不是概念验证（demo / POC）**，而是一款 **领域深度较高、工程纪律较强、产品外壳尚未完备** 的桌面创作工具。其当前位置更接近：

> **工程化的高级 Beta / 可内测的产品内核（Product Core）**

而非「可运营、可大规模对外宣称的正式商业 1.0」。

### 1.2 一句话定位

| 维度 | 判断 |
|------|------|
| 相对 demo | **明显偏右**：存在正式文件格式、保存语义、领域不变量、CI 与 Agent 协议 |
| 相对正式产品 | **仍偏左**：发布运维、跨平台、E2E 验收、设置与权限产品化不足 |
| 综合坐标 | **工程原型右缘 → 高级 Beta 左缘** |

### 1.3 成熟度评分（1–10）

| 维度 | 评分 | 说明 |
|------|-----:|------|
| 领域实现深度 | **8** | 画布、嵌套栈、媒体、保存、Agent/MCP 均具备真实产品复杂度 |
| 工程纪律 | **7** | TypeScript 分层、单元测试、CI typecheck 完备；缺 lint/E2E/覆盖率门槛 |
| 发布与运维就绪度 | **4** | 具备 Windows 打包目标与 portable 产物，缺签章、自动更新与正式发版流水线 |
| 产品外壳完整度 | **4–5** | 核心创作链路可用；设置、i18n、权限模型、崩溃运营体系薄弱 |
| **综合产品化就绪度** | **5** | 适合内测/小范围分发；不宜直接定义为大规模正式版 |

## 2. 评估方法

### 2.1 依据来源

本评估仅基于仓库内可观察的工程事实，包括但不限于：

- 源码结构与模块边界（`src/`、`src-tauri/`、`packages/ic2-mcp/`）
- 构建与质量脚本（`package.json`、`vite.config.ts`、GitHub Actions）
- 桌面打包配置（`src-tauri/tauri.conf.json`、`Cargo.toml`）
- 运行时安全与诊断实现（CSP、SSRF 防护、ErrorBoundary、diag log）
- 自动化测试规模与通过情况

**不纳入结论依据**：`docs/` 内产品/架构说明的叙事内容（避免文档与代码脱节带来的乐观偏差）。

### 2.2 评估框架

采用四层模型：

1. **领域层（Domain）**：核心业务是否可长期演进  
2. **工程层（Engineering）**：可维护性、可回归性、模块化  
3. **交付层（Delivery）**：构建、安装、更新、平台覆盖  
4. **产品层（Productization）**：设置、权限、可观测、规模化体验

## 5. 产品化缺口（为何尚非正式版）

### 5.1 交付与运维层

| 缺口 | 证据 | 风险 |
|------|------|------|
| 平台覆盖窄 | 打包 targets 仅 Windows 安装器形态 | 无法作为跨平台正式产品宣称 |
| 无自动更新 | 未见 updater 插件与更新通道 | 缺陷修复依赖用户重装，运营成本高 |
| 发版流水线不完整 | CI 仅 check，不构建/签章/上传 Release | 发布不可重复、不可审计 |
| 代码签名缺失（工程未见） | 无签名流程配置 | Windows SmartScreen 劝退，分发信任不足 |
| 版本叙事不一致 | 主应用 `1.1.0`，MCP `0.1.0`，`private: true` | 对外版本语义不清晰 |

### 5.2 质量门禁层

| 缺口 | 证据 | 风险 |
|------|------|------|
| 无 E2E | 工程脚本无 Playwright/Cypress 主路径 | 指针手势、拖拽、栈动画等主交互依赖人测 |
| 无覆盖率门槛 | CI 未强制 coverage | 重构时回归保障不可量化 |
| 无统一 lint/format 体系 | 无项目级 ESLint/Prettier 依赖（仅零星注释） | 风格与隐患依赖个人习惯 |
| 热点巨型模块 | 单文件可达 ~900–1400 行（手势、栈退出、writeExtras、主画布） | 变更耦合高、评审困难、缺陷密度上升 |

代表性热点文件（行数量级，评估时点）：

| 文件 | 约行数 | 风险类型 |
|------|-------:|----------|
| `src/hooks/canvas/useCanvasPointerGestures.ts` | ~1429 | 交互主路径过于集中 |
| `src/board-ops/writeExtras.ts` | ~1175 | Agent/写路径膨胀 |
| `src/store/actions/stackExitNavigation.ts` | ~1054 | 栈语义高风险区 |
| `src/components/InfiniteCanvas.tsx` | ~935 | 渲染与状态交织 |
| `src/utils/stacks.ts` / `linkMeta.ts` | ~900+ | 工具层过载 |

### 5.3 产品外壳层

以下能力在代码中基本缺失或仅有雏形：

| 能力 | 现状 | 对「正式版」的影响 |
|------|------|-------------------|
| 用户设置 / 偏好持久化 | 几乎仅 diag/perf 使用 localStorage | 无法形成可配置产品体验 |
| 国际化（i18n） | 未见 | 限制受众与专业化观感 |
| 无障碍（a11y） | 局部 `aria-*`，非系统策略 | 画布应用可降级要求，但仍缺产品声明与基线 |
| 崩溃上报 / 远程可观测 | 仅本地日志 | 无法量化真实崩溃率与保存失败率 |
| 账户 / 云同步 / 协作 | 无 | 若路线图包含协作，当前为零基础 |
| Agent 权限模型 | `allowAgentWrite: true` 硬编码倾向 | 安全默认值不符合正式产品原则 |

### 5.4 架构取舍带来的规模上限

| 设计选择 | 收益 | 正式版压力 |
|----------|------|------------|
| 媒体 base64 内嵌 `.icanvas` | 离线、可搬运、路径不失效 | 大板子易至数百 MB，WebView 内存与保存时延承压 |
| 业务逻辑主要位于前端 | 迭代快、调试友好 | 大文件 I/O、重计算未充分下沉原生层 |
| 本地文件式 Agent 桥 | 实现简单、可与 MCP 协同 | 多实例、权限、审计、稳定性仍偏实验 |
| 薄 Rust 命令面 | 边界清晰 | 原生能力未形成性能与安全护城河 |

## 6. 与「可发布正式版」的差距分析

### 6.1 目标定义

建议将「正式版」拆为可验收的两个里程碑，避免范围失控：

| 里程碑 | 定义 | 建议定位 |
|--------|------|----------|
| **M1 — Windows 正式 1.0** | 可签名安装、主路径可自动验收、大板子可用、Agent 默认安全 | **最近可达的正式版** |
| **M2 — 多平台 / 可运营产品** | macOS（及可选 Linux）、自动更新、设置与 i18n、可观测运营 | 1.0 之后 |

### 6.2 差距清单（按优先级）

#### P0 — 发布可信度（Blocking for GA）

| ID | 工作项 | 完成判据 |
|----|--------|----------|
| P0-1 | 可重复的 Windows 发版流水线 | CI/CD 产出 nsis/msi/portable，版本号与 changelog 一致 |
| P0-2 | 代码签名与 SmartScreen 策略 | 安装/首次启动无明显拦截或有官方信任路径说明 |
| P0-3 | 主路径 E2E 冒烟 | 打开 / 保存 / 导入媒体 / 撤销 / 进退栈 / 关闭未保存 自动化通过 |
| P0-4 | 大文件与内存退化策略 | 超大板有进度、失败可恢复、明确上限与用户提示 |
| P0-5 | 保存失败与关闭路径验收 | 失败不丢数据意图；关闭拦截与队列语义有自动化覆盖 |

#### P1 — 可维护性与产品壳（Should for GA）

| ID | 工作项 | 完成判据 |
|----|--------|----------|
| P1-1 | 拆分巨型交互/栈/写路径模块 | 热点文件下降到可评审单元；关键路径有单测/契约测 |
| P1-2 | ESLint + 格式化 +（可选）覆盖率门槛 | PR 强制通过；禁止无说明的 `any` 扩散 |

| P1-4 | 更新机制 | 至少「检查更新」；理想状态自动更新通道 |
| P1-5 | 诊断导出产品化 | 一键导出 diag + 环境信息，支持支持工单 |

#### P2 — 增长与平台化（Post-GA）

| ID | 工作项 | 完成判据 |

| P2-1 | macOS（及可选 Linux）构建矩阵 | 同版本多平台产物 |

| P2-5 | 可选遥测 | 崩溃率、保存耗时分位等聚合指标（需隐私合规） |

### 6.3 建议的最短正式版路径

将资源收敛到以下公式：

```text
Windows 正式 1.0
  = 现有产品内核
  + 可重复发版与代码签名
  + 主路径 E2E
```

## 9. 最终结论

### 9.1 项目属性

- **属性**：工程化桌面创作工具内核（Engineered desktop tool core）  
- **阶段**：高级原型 / 高级 Beta 前夜  
- **可用性**：适合 **内测、个人生产力、Windows 小范围分发**  
- **不宜宣称**：跨平台正式版、企业级可运营产品、大规模零支持自助分发

### 9.2 距离正式版的本质差距

差距主要不在「会不会做画布功能」，而在：

1. **交付可信度**（签章、可重复发版、更新）  
2. **主路径质量保证**（E2E 与热点模块治理）  
3. **规模化体验**（大文件、内存、失败恢复）  
4. **产品默认安全与可配置性**（尤其 Agent 写权限）

### 9.3 决策建议

| 目标 | 建议 |
|------|------|
| 尽快对外 | 以 **Windows 签名 1.0** 为唯一 GA 目标，冻结跨平台与云协作 |
| 继续打磨内核 | 优先拆分手势/栈/写路径，并用 E2E 锁主路径 |
| 对外讲故事 | 定位为「专业向无限画布 + Agent 可编程」，避免「全平台正式产品」表述 |
